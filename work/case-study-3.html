Case Study - 03

Design system i frontend dla platformy B2B - od chaosu do 150+ komponentow

Rozwijajacy sie startup B2B mial problem typowy dla etapu wzrostu: 4 lata developmentu bez jednorodnego systemu UI. Kazdy ekran wygladal troche inaczej. Onboarding trwal 3 tygodnie. Czas na budowe nowej funkcji - 2-4 x dluzej niz powinnien.

Klient
Produkt B2B, SaaS
Zakres
UI Audit, Design System, Frontend
Czas realizacji
12 tygodni
Rok
2024
platforma-b2b-ui-system.jpg

Problem

Produkt rozwijany przez 2 dev-teams bez wspolnego wzorca. 6 roznych rodzajow przycisku w jednej aplikacji. 4 rozmiary fontu, ktore nie mialy logiki. Nowy developer potrzebowal tygodnia, zeby zrozumiec, ktory komponent gdzie uzywac. Klientow grzebalo sic w interfejsie.

Podejscie

Audyt istniejacego UI (pełna inwentaryzacja komponentow), potem projekt systemu od tokenow, nie od komponentow. Tokens → primitives → semantics → components → patterns. Kazdy etap zatwierdzany przez lead developer i produkt owner zanim ruszalem do implementacji.

Audyt i inwentaryzacja

Tydzien pracy: przejrzalem 40+ ekranow produkcyjnych, skatalogowałem wszystkie unikalne warianty kazdego komponentu. Raport: 47 wariantow przycisku, 12 sposobow wyswietlania daty, brak jednorodnego systemu odstepow. Dopiero po tym audycie moglem zaproponowac, ile z tego utrzymujemy, a ile wyrzucamy.

"Design system nie jest projektem graficznym. To infrastruktura produktu."

Tokens jako fundament

Zamiast projektowac przyciski, zaprojekowalem zmienne. Kolory, przestrzenie, promienie, cienie, typografia - wszystko jako named tokens z jasna semantyke. Potem prymitywy zgrupowane w semantyczne aliasy (primary, danger, surface-elevated). Od tej chwili Developer nie decyduje: "jaki kolor przycisk hover?" - system decyduje. Zmieniasz token, zmieniaja sie wszystkie wdrozenia.

Biblioteka komponentow (Figma + kod)

150+ komponentow. Kazdy w Figmie z properties i variantami mapujacymi sie 1:1 na props w kodzie. Implementacja jako czysty HTML/CSS/JS (bez frameworka - wymog klienta), z eksportem do Storybook. Kazdy komponent z dokumentacja, przykladami uzycia i stated (default, hover, focus, disabled, error).

Wdrozenie i migracja

Migracja rozlozowana na 3 sprinty. Komponenty wprowadzane stopniowo przez feature flags - bez single big-bang release. Zerow regresji wykrytych przez klientow na produkcji. Plik README i Storybook z przykladami ulatwily onboarding nastepnych developerow z 3 tygodni do 3 dni.
